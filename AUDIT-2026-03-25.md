# AGENTBOT PLATFORM AUDIT REPORT
**Date:** 2026-03-25
**Auditor:** Atlas (PAI Comprehensive Mode — 6 parallel agents + direct code review)

## EXECUTIVE SUMMARY

**Overall Status: 🟡 FUNCTIONAL WITH CRITICAL GAPS**

The security foundation is solid — auth, crypto, SSRF, webhooks are correctly implemented. However, three critical functional gaps exist that would cause runtime failures for paying users: the job queue is entirely non-functional, the agent clone feature returns fake success, and the plan enforcement middleware has inverted plan definitions causing wrong model/agent access.

---

## 🔴 CRITICAL ISSUES (Fix Before Next Release)

### C-1: BullMQ Worker — ALL Handlers Are Stubs
**File:** `agentbot-backend/src/worker.ts:16-57`
**Impact:** Scheduled tasks, skill execution, and async agent provisioning all fail at runtime.

Every job handler throws `Error('not yet implemented')`:
```ts
// worker.ts:28
throw new Error(`scheduled-task handler not yet implemented (taskId: ${job.data.taskId})`);
// worker.ts:32
throw new Error(`skill-execution handler not yet implemented (skill: ${job.data.skillName})`);
// worker.ts:51
throw new Error(`new-agent handler not yet implemented (userId: ${job.data.userId})`);
```

Jobs queued to `tasks` or `provision` queues will immediately fail. Any feature relying on background job processing is silently broken.

**Fix:** Implement the three handlers or remove them and document what queues are in use.

---

### C-2: Plan Enforcement Middleware — Inverted Plan Definitions
**File:** `agentbot-backend/src/middleware/plan.ts:4-33`
**Impact:** Model access control (`routes/ai.ts`) enforces wrong limits. Users on expensive plans get fewer resources than cheap plans.

| Plan | plan.ts agents | Correct (pricing.ts) | plan.ts price | Correct price |
|------|----------------|----------------------|---------------|---------------|
| `label` | 1 agent | 10 agents | $29 | £149 |
| `solo` | 3 agents | 1 agent | $79 | £29 |
| `collective` | 10 agents | 3 agents | $199 | £69 |
| `network` | 100 agents | unlimited | $499 | £499 |

Label (£149) users get 1 agent and 3 models. Solo (£29) users get 3 agents and 10 models. **This is backwards.**

`requirePlan` + `canAccessModel` are actively used in `routes/ai.ts:72,93,112`. This is live enforcement.

**Fix:** Align `plan.ts` PLANS object with `web/app/lib/pricing.ts` as the source of truth.

---

### C-3: SQL Column Mismatch — Royalty Splits Broken
**Files:** `underground.ts:138` vs `services/db-init.ts:80`
**Impact:** Any royalty split execution throws a PostgreSQL error.

DB schema defines: `royalty_recipients.percentage`
Insert query uses: `royalty_recipients.share_percentage` (doesn't exist)

```ts
// underground.ts:138 — WRONG column name
'INSERT INTO royalty_recipients (split_id, wallet_address, share_percentage, paid) ...'
// db-init.ts:80 — actual column
percentage NUMERIC NOT NULL,
```

**Fix:** Change `share_percentage` → `percentage` in `underground.ts:138`.

### C-4: Agent Clone — Returns Fake Success
**File:** `web/app/api/agents/clone/route.ts:41-44`
**Impact:** Clone endpoint accepts payment, generates wallet, returns `status: "provisioning"` — but never stores to DB, never spawns container, never sets up revenue sharing.

```ts
// TODO: Store in database
// TODO: Spawn Docker container for the agent
// TODO: Set up parent-child revenue sharing
// TODO: Update colony fitness scores
```

Users who pay to clone an agent get nothing. Payment is collected but feature doesn't work.

**Fix:** Implement or disable the endpoint. If disabling, return 501.

---

### C-5: RISC Endpoint — JWT Signature Not Verified
**File:** `web/app/api/security/risc/route.ts:98-101`
**Impact:** Google security events (account compromise, token revocation) could be spoofed.

```ts
// For production, verify the signature using Web Crypto API
// For now, we trust the token if issuer and audience are valid
// TODO: Full RSA signature verification
```

The endpoint checks `iss` and `aud` claims but does NOT cryptographically verify the JWT signature. An attacker could forge a RISC event to revoke sessions.

**Fix:** Implement RS256 signature verification using the fetched Google JWK keys.

## 🔍 ADDITIONAL FINDINGS (Backend Routes Agent)

### M-7: `/api/heartbeat` — Unauthenticated Activity Spoofing
**File:** `routes/registration.ts:166`
**Impact:** Any caller can POST `{ userId: "any_id" }` to set `status='active'` and `last_seen=NOW()` for any user in `agent_registrations`.

```ts
// No auth middleware — public endpoint
router.post('/heartbeat', async (req, res) => {
  pool.query(`UPDATE agent_registrations SET last_seen = NOW(), status = 'active' WHERE user_id = $1`, [userId])
})
```

Not a data-leak but activity data is untrustworthy and any agent can be spoofed as alive.
**Fix:** Add `authenticate` middleware, or require the caller to prove ownership via Bearer token.

---

### M-8: `/api/invite/validate` — 48-bit Entropy, No Rate Limit at Route Level
**File:** `invite.ts:43`
**Impact:** Invite code validation is unauthenticated and codes are 12-char hex (48-bit). The general 120 req/min global limiter provides minimal brute-force protection.

**Fix:** Add a tighter per-IP rate limit on `/api/invite/validate` (e.g. 5/min). Consider upgrading to 32-char hex (128-bit).

### CORRECTION: agents.ts, openclaw.ts, render-mcp.ts — Auth Confirmed at Mount Level
The routes agent reported 0/15 auth coverage on agents.ts. **This is a false positive.** All three routers are wrapped with `authenticate` at index.ts mount:

```ts
app.use('/api/agents',      authenticate, agentsRouter);   // line 710
app.use('/api/openclaw',    authenticate, openclawRouter);  // line 711
app.use('/api/render-mcp',  authenticate, renderMcpRouter); // line 707
```

Route-level auth is absent but mount-level auth gates them correctly. No action needed.
